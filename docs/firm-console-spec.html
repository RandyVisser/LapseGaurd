Firm Console

One place where a property-management firm runs its whole book: the associations it manages, the people who manage them, and how it all gets billed. Scales from a one-person shop to a 1,000-PM company without changing shape.

Route /admin/firm (replaces current page) Audience firm owners · managers · super users Status proposal — Randy & Troy to review Date 2026-07-12

The shape: five tabs, three roles

Everything firm-level lives under one console with tabs. What you can see and do is decided by your role in the firm — the Active-Directory idea: users and groups are managed in one place, and access to resources (associations) flows from membership.

TAB 1 · OVERVIEW — the portfolio at a glance. KPI row, the same needs-attention language as the association dashboard, and a “lowest compliance first” list, because the worst building is the one a managing member acts on. Every number click-throughs to the filtered view that produced it.

Scoping rule: the console shows you what your role can see. An owner or manager sees the whole firm; a member in an assignment-based firm sees the same tabs computed over their book only. One design, no special cases.

TAB 2 · ASSOCIATIONS — the registry. Sortable, searchable, paginated past ~25 rows. Avatar chips are editable in place (owners/managers): click to assign PMs to that association — the inverse of editing a person. “Unassigned” renders as a warning state because in an assignment-based firm it means nobody is watching that building. “Open ▸” jumps into the existing per-association dashboard.

Add association — the firm front door

The button on this tab is the missing self-serve flow: the firm creates the association (name + address), imports units with the existing wizard, and the association lands in the portfolio assigned to whoever added it — a new PM literally brings their book. Optionally invite a board contact by email; the board logging in is optional, exactly how a PM-run association works in practice. Billing follows the firm's mode automatically: firm-paid associations fold into the next quantity sync, pass-through ones get the firm rate.

TAB 3 · PEOPLE / GROUPS — Troy's Green/Yellow/Red sketch, literally. A group is a named set of PMs and a set of associations; putting a person in a group grants them the group's whole book. “Moving someone around” = changing group chips, one click, instead of re-ticking 30 checkboxes. A person can be in several groups (Dana covers Green and Red), and direct assignments still work for one-off coverage.

* In an open-visibility firm every member sees everything — the small-firm default, unchanged. The matrix's “assigned only” column applies once the owner turns on assignment-based visibility. Both of Randy's open calls are taken as the recommendation here: members can add associations (self-assigned), and managers get read-only billing.

One access rule, extended once

All of this resolves through the single chokepoint that already exists (services/firms.py). The rule grows one clause for groups and one for managers:

member may access association ⇔ firm manages it AND ( firm is open-visibility OR role ∈ {owner, manager} OR directly assigned OR shares a group that includes it )

Data & API deltas

pm_firm_members.role        text: 'owner' | 'manager' | 'member'   (migrate is_owner → role)
pm_groups                   id, firm_id, name, color
pm_group_members            group_id, supabase_user_id     (FK → pm_firm_members, cascade)
pm_group_hoas               group_id, hoa_id               (FK → pm_firm_hoas, cascade)

GET  /pm/overview           aggregated KPIs + worst-compliance list (ONE server-side
                            query pass — this finally builds the at-scale aggregation)
GET  /pm/associations       registry rows: units, status counts, assigned PMs, billing
POST /pm/associations       firm-side “Add association” (create + map + assign creator)
PATCH /pm/team/members/{id} role changes (owner only)
CRUD /pm/groups             groups + membership + association sets (owner/manager)

Billing scope is untouched: subscriptions, webhook fan-out, and the nightly quantity sync keep reading the firm portfolio (pm_firm_hoas), never assignments or groups. Groups change what people see, never what gets billed.

Build order

~1 day

Phase 1 — Console + roles + registry

  • Tab shell at /admin/firm; current Team/Billing panels become the People and Billing tabs
  • role migration (owner/manager/member) + permission matrix enforcement
  • Associations tab with GET /pm/associations; Overview tab with GET /pm/overview
  • Super users get the same console per firm from their directory (read-only)
~half day

Phase 2 — Add association + invite polish

  • Firm-side “Add association” (create → import wizard → auto-assign → optional board invite)
  • Invite PM with pre-assigned associations
~1 day

Phase 3 — Groups

  • Groups schema + the one-clause access-rule extension + Groups sub-tab
  • Assignment side panel gains “add to group” and “copy from person”

Decisions locked in this spec (veto anytime)

1 · Members can add associations, self-assigned. The “bring your book” flow beats funneling everything through managers; managers can always reassign after.

2 · Managers get read-only billing. A regional director should see what the portfolio costs without being able to cancel the firm's subscription.

3 · Groups are additive, not exclusive. Effective access = direct assignments ∪ group books. No “primary group”, no hierarchy of groups — flat, like AD security groups, because hierarchy is where these systems go to die.

4 · The console replaces the current /admin/firm page rather than adding a second place. Settings keeps only per-association concerns.
